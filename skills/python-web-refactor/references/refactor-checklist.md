# Python Web 服务重构检查清单

## 事实收集

- 是否有 README、接口文档、配置说明、架构说明
- 实际启动入口有哪些，哪个才是主入口
- 是否存在脚本、定时任务、后台任务与 API 混写
- 配置来源有哪些：环境变量、配置文件、数据库、硬编码
- 是否已有测试，测试主要覆盖什么链路

## 进程与边界

- 当前是否真有长任务、重 IO、外部调用编排
- 这些任务是否直接跑在请求线程里
- 是否需要拆出 worker / scheduler，还是只需要先把服务层拉直
- 入口是否只做装配，还是堆了大量初始化和业务逻辑

## 目录与模块

- 路由层是否只做协议适配
- 业务逻辑是否散落在 `router/model/script`
- 服务层是否承载跨依赖、跨模型流程
- 一次性工具、迁移脚本、运维脚本是否独立
- 模块命名是否按职责而不是按历史阶段堆叠

## 配置治理

- 环境变量是否可枚举、可说明
- 运行时行为开关是否与进程级配置分离
- 是否存在硬编码 token、路径、第三方地址
- 启动脚本是否统一加载 `.env` / `.venv` / `PYTHONPATH`

## 接口与数据

- 现有 API 边界是否清楚
- 请求体、响应体、状态码是否稳定
- 模型层是否混入协议层字段
- 是否先有文档和代码依据，再决定接口行为

## 测试与验证

- 是否有覆盖主链路的集成测试
- 是否只因为测试方便而过度 mock
- 重构后是否还能验证入口、配置、主链路和关键异步流程
- 失败场景和回归风险是否有对应验证

## 文档与治理

- 是否有统一文档入口
- 愿景、架构、接口、配置是否分开维护
- 是否有明确的重构阶段、验收标准、风险与回滚点
- 当前仓库若要求方案阶段产出 issue / ADR / 任务单，是否已纳入流程
