# Job System Design Principles

## 1. Single Source of Truth

- 使用统一的任务存储承载任务生命周期状态与执行结果，不要把真实执行状态只藏在进程内存里。
- 任务记录至少应包含：`type/name`、`status`、`params`、`created_at/started_at/finished_at`、`progress`、`error/stack`、`worker_id`（可选）。

## 2. Web/API 与执行面解耦

- Web/API 进程只负责创建任务、查询任务和有限的运维操作。
- 不要在请求线程内执行重活，避免超时、阻塞和不可控资源占用。

## 3. 多 Worker + 任务拆分优先

- 并发优先来自多个独立 worker（进程或容器实例）。
- 大任务优先拆分为多个子任务表达并行度，而不是在单个任务函数里再开线程池或进程池。
- 任务内部嵌套并发会降低可观测性，并放大资源竞争问题。

## 4. Atomic Claim / Compare-and-Set

- Worker 领取任务必须是原子操作，只允许一个 worker 成功执行 `pending -> running`。
- 到期任务和普通任务应有清晰的领取策略，避免未来任务阻塞当前任务。
- 领取逻辑通常需要结合 `scheduled_for`、状态过滤和抢占锁或条件更新实现。

## 5. 外部依赖门控（Concurrency / Rate Guardrails）

- 将外部系统的并发上限、连接数上限、速率限制和风控策略内建为显式门控。
- 门控可以通过信号量、令牌桶、互斥锁或分布式锁实现。
- 默认配置应以稳定、可复现为优先；提升并发必须显式配置，而不是隐式放开。

## 6. 资源生命周期最小化

- 任务执行前后再打开和关闭关键资源，尤其是 DB 连接与事务。
- 避免长任务长期持有连接，导致连接池耗尽。
- 读取任务元数据与执行重活应在资源占用上解耦，重活期间不占用不必要资源。

## 7. 幂等与可重跑

- 默认假设任务可能被重试、重复投递或执行中断，写入侧应尽量幂等。
- 常见做法包括 upsert、唯一键、幂等键和版本号。
- 需要明确幂等锚点，例如时间窗口、批次号或任务参数签名，并确保重复运行得到一致结果。

## 8. Parent-Child 编排与进度聚合

- 对可拆分任务使用父子任务建模：父任务负责汇总，子任务承载实际计算。
- 父任务进度聚合应可解释，至少能看出完成数、失败数、运行中数和总数。
- 进度模型需要直接服务 UI 展示和排障，而不是只为内部实现方便。

## 9. 故障恢复与状态回收

- 任务失败必须显式记录失败原因，包括可定位的错误信息和堆栈。
- 对 `running` 卡死任务要有回收机制，例如超时、心跳或 lease 到期后回到可重试状态。
- 回收动作应记录原因。
- 重试策略必须可控，包括哪些错误可重试、重试间隔、最大次数和最终去向（如 dead-letter）。

## 10. 可观测性与脱敏

- 结构化日志应包含 `job_id/type/status/attempt/worker_id` 等关联字段，便于聚合检索。
- 指标建议覆盖吞吐、排队延迟、执行时长、失败率、重试次数和外部依赖调用统计。
- 时间维度视图应支持按时间桶聚合 pending/running/done 的数量与类型分布。
- 日志和错误信息默认不包含密钥、token、密码或完整敏感请求体。
