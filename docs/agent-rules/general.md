# 通用协作与编码细则

## 基本原则

- 先看证据，再动代码；证据包括文档、代码、日志、测试和可复现步骤。
- 不要把“猜测的接口行为”写进实现；没有依据时先补上下文。
- 修 bug 必须定位根因；禁止仅通过额外分支、吞异常或默认值掩盖故障。
- 优先做简单、可验证、可维护的实现，避免为了“可能的未来需求”过度设计。

## 代码改动原则

- 沿着现有结构改，不随手引入新的框架层或目录体系。
- 修改路由时，同时检查对应模板、静态资源、服务函数和测试是否受影响。
- 修改数据库读写时，以 `db.py` 中的 schema 和接口为准；不要让文档或模板反向定义数据结构。
- 修改试卷解析或试题生成时，同时核对 `qml.md`、`qml/parser.py` 和相关测试。
- 涉及 LLM、短信、号码认证时，优先保证“未配置时失败可解释、已配置时行为清晰”。

## 测试与验证

- 能跑测试就跑与改动最相关的测试，不做空口确认。
- 修复行为回归时，优先补回归测试。
- 对外部服务依赖较强的逻辑，优先验证本地可退化路径是否仍可工作。

## 文档维护

- 新增专题知识时，优先写进 `docs/`，不要把实现细节塞进根 `README.md`。
- 文档只写当前代码能证明的事实；不写“计划中的结构”当成现状。
- 路径变更、模块拆分、配置项变化后，及时更新 `docs/README.md` 导航和相关专题页。

## 安全与数据

- 不在回复、日志或截图中暴露密钥、候选人手机号、简历原文等敏感信息。
- 不执行破坏性 Git 命令，不覆盖用户未授权的本地改动。
